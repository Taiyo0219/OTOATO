# OTOATO

「その場所、その時間に聴いていた一曲を残す」位置情報音楽共有Webアプリです。

YouTubeから一曲を選び、場所・時間・コメントと一緒に投稿できます。投稿した曲は地図や日付から振り返ることができ、相互フォローしたユーザーの投稿や、その日にどこでどんな曲を残したかも確認できます。

## 概要

OTOATOでは、曲を検索し、現在地または地図上で選んだ場所に投稿できます。投稿にはコメントと公開範囲を付けられます。

ホーム画面では地図上の投稿を見られます。アーカイブ画面では日付を選び、その日に残された曲を振り返れます。

ユーザー検索、フォロー、相互フォローにも対応しています。相互フォローしたユーザーは友達として扱い、友達限定の投稿を見られます。また、「音の足あと」では、ユーザーと日付を選び、その日に投稿された曲を時系列と地図で確認できます。

## 制作背景

写真を撮っていない出来事でも、後から当時聴いていた曲を聞くと、その場所や気分を思い出すことがあります。

そこで、「その時に聴いていた一曲」を場所と時間に結びつけて残せないかと考えました。

OTOATOでは、再生履歴を自動で共有するのではなく、ユーザーがその場で一曲を選んで投稿します。写真投稿を中心にしないことで、音楽を記憶の中心に置きつつ、位置情報と写真を同時に公開することへの不安も減らせると考えています。

## 主な機能

### 1. ホーム画面・地図

ホーム画面では地図を中心に表示します。現在地を取得すると、周辺に残された投稿を地図上のマーカーで確認できます。

マーカーをタップすると、曲名、アーティスト、コメントなどが表示されます。そこから投稿詳細を開くこともできます。

<!-- ここにホーム画面・地図画面のスクリーンショットを貼る -->

### 2. 曲を検索して投稿

投稿画面では、YouTube Data API v3を使って曲を検索します。検索は入力中に毎回実行せず、検索ボタンまたはEnterキーで実行します。

検索結果から曲を選び、コメントと公開範囲を設定して投稿します。公開範囲は「公開」「友達」「自分だけ」から選べます。

<!-- ここに曲検索から曲選択までが分かる投稿画面のスクリーンショットを貼る -->

### 3. 投稿地点の選択

投稿地点は、現在地取得または地図上のタップで決められます。

位置情報が拒否された場合でも、地図から場所を手動で選べるため、投稿フローが止まらないようにしています。位置情報は必要なときだけ取得し、常時追跡は行いません。

<!-- ここに投稿地点を地図から選択している画面のスクリーンショットを貼る -->

### 4. 日付から投稿を振り返る

アーカイブ画面では、日付を選んでその日の投稿を表示できます。曲、コメント、投稿時刻を見ながら、過去の音楽の記録を振り返れます。

<!-- ここにアーカイブ画面のスクリーンショットを貼る -->

### 5. ユーザーを探す

探す画面では、表示名でユーザーを検索できます。検索結果には、プロフィール文、好きなジャンル、フォロー状態が表示されます。

検索結果からプロフィールを開き、気になるユーザーをフォローできます。

<!-- ここにユーザー検索結果が表示されている「探す」画面のスクリーンショットを貼る -->

### 6. フォローと友達限定公開

AがBをフォローし、BもAをフォローすると、友達として判定します。

投稿時に「友達」を選んだ投稿は、投稿者本人と相互フォローしているユーザーだけが閲覧できます。フロントエンドでボタンを隠すだけでなく、API側でも閲覧権限を判定しています。

<!-- ここにフォロー状態または友達状態が分かるプロフィール画面のスクリーンショットを貼る -->

### 7. プロフィール・マイページ

マイページでは、表示名、自己紹介、好きなジャンルを編集できます。自分の投稿、フォロー中のユーザー、フォロワーも確認できます。

他ユーザーのプロフィールでは、その人のプロフィール情報と、自分が閲覧できる投稿だけを表示します。

<!-- ここにマイページまたはプロフィール画面のスクリーンショットを貼る -->

### 8. 音の足あと

ユーザーと日付を選ぶと、その日に投稿された曲を時系列順に表示します。地図上の投稿地点には1、2、3の番号を付け、下の曲カードと対応させています。

これにより、どの場所でどの曲を残したのかを順番に確認できます。

これは実際のGPS移動経路ではありません。自動で移動履歴を保存するのではなく、ユーザーが投稿した地点を投稿時刻順に並べています。

## UI/UXで意識した点

- スマートフォンで使うことを前提に、下部ナビゲーションを「ホーム / 探す / 投稿 / マイページ」の4項目にしました。
- 未ログインでも地図、投稿閲覧、曲検索、アーカイブは使えます。投稿するときだけログインを求めます。
- 投稿途中でログイン画面へ移動しても、選択中の曲、地点、コメントを復元できるようにしています。
- 現在地が取得できない場合でも、地図から投稿地点を選べます。
- 投稿時の公開範囲には「誰でも見られます」「相互フォローだけ」など短い説明を付けました。
- 「音の足あと」では、地図上の番号と曲カードの番号を対応させました。
- 実際の移動経路と誤解されないよう、投稿地点同士を線で結ばず、番号付きマーカーで表示しています。

## 位置情報とプライバシーについて

位置情報は `getCurrentPosition` を使い、必要なときだけ取得します。`watchPosition` は使っていません。

常時位置追跡は行わず、移動履歴も自動保存しません。投稿する場所は、現在地取得または地図上の手動選択でユーザーが決めます。

投稿には `public`、`friends`、`private` の公開範囲を設定できます。写真投稿を主機能にしていないことも、位置情報を扱うアプリとしての方針の一つです。

  ↓ REST API

## セットアップ方法

1. リポジトリを取得します。

```bash
git clone https://github.com/Taiyo0219/OTOATO.git
cd OTOATO
```

2. パッケージをインストールします。

```bash
npm install
```

3. 環境変数を設定します。

`server/.env.example` と `client/.env.example` を参考に、必要な `.env` を作成します。

4. アプリを起動します。

```bash
npm run dev
```

起動後のURL:

- Frontend: http://localhost:5173
- Backend: http://localhost:5000

Windows PowerShellで `npm` が実行ポリシーにより止まる場合は、`npm.cmd` を使用します。

```powershell
npm.cmd install
npm.cmd run dev
```

## 環境変数

秘密情報をGitへcommitしないでください。

CLIENT_ORIGIN=http://localhost:5173

```env

ローカル開発では `VITE_API_BASE_URL` は未設定のまま使用します。本番ではRenderのURLを `/api` なしで指定します。

```

## 実装上の工夫

### 公開範囲をサーバー側で判定

`public`、`friends`、`private` の閲覧制御はAPI側で行っています。

周辺投稿、アーカイブ、投稿詳細、他ユーザー投稿、音の足あとで同じ認可ロジックを使い、画面によって非公開投稿が見えてしまうことを防いでいます。

### Followを別コレクションで管理

フォロー関係はUser内の配列ではなく、Followコレクションで管理しています。

`followerId` と `followingId` の組み合わせにunique indexを設定し、重複フォローを防いでいます。相互フォローかどうかも、このFollowデータから判定しています。

### JWTによる認証

VercelとRenderを分離した構成でも扱いやすいように、認証にはBearer Token方式のJWTを使っています。

現在はクライアント側の `localStorage` にトークンを保存し、APIリクエスト時に `Authorization` ヘッダーへ付与しています。今後は、より安全な保存方法としてhttpOnly Cookie方式も検討しています。

### 既存匿名投稿との互換性

ユーザー機能を追加する前の投稿には、投稿者情報が存在しない可能性があります。

そのような投稿も壊さず表示できるよう、投稿者がいない投稿は既存の公開投稿として扱うようにしています。

### 日本時間の日付検索

MongoDBにはUTCで日時を保存しています。

日付検索では、日本時間の1日をUTC範囲に変換し、サーバー環境のタイムゾーンに依存しないようにしています。

### ローカルと本番でAPI接続先を切り替える

開発環境ではVite Proxyを使って `/api` をExpressへ転送します。

本番環境では `VITE_API_BASE_URL` を使い、VercelからRenderのAPIへ通信します。

## 開発中に苦労した点

### スマートフォンからAPIへ接続できない問題

最初はAPI接続先を `localhost:5000` に固定していたため、スマートフォンからアクセスするとスマートフォン自身のlocalhostを見に行ってしまいました。

開発環境ではVite Proxyで `/api` を使い、本番環境では `VITE_API_BASE_URL` を使う構成に変更しました。

### 位置情報が取得できない問題

LAN内のHTTP URLでは、スマートフォンのGeolocation APIが使えない場合がありました。

本番環境をVercelでHTTPS化し、あわせて地図から手動で投稿地点を選べるようにしました。

### MongoDB Atlasを本番環境から接続する問題

ローカルでは接続できても、RenderからMongoDB Atlasへ接続するには、Atlas側のNetwork Access設定が必要でした。

接続設定、環境変数、Render側の起動設定を分けて確認し、本番環境でもMongoDBへ接続できるようにしました。

### 公開範囲の認可

友達限定投稿や自分だけの投稿は、フロントエンドだけで制御するとAPIから直接取得できてしまう可能性があります。

そのため、投稿取得API側で共通のvisibility条件を使い、どの画面でも同じルールで取得するようにしました。
